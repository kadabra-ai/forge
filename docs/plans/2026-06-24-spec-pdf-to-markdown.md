# Spec PDF → high-fidelity Markdown (Marker / Docling)

Date: 2026-06-24
Status: Ready to start (own session)
Goal: replace the hand-made, lossy `docs/spec/*.md` with high-fidelity Markdown + extracted
figures, generated by a layout-aware tool, so both the user and Claude can use the OMG specs as a
searchable reference (diagrams preserved, code blocks fenced correctly).

## Why (context)

- The big spec PDFs are in `vendor/SysML-v2-Release/doc/` (KerML, SysML, API, two intros). 454-page
  KerML spec is the priority.
- The current `docs/spec/*.md` are manual `pdf-to-markdown` conversions: diagrams lost, code blocks
  mangled. Painful to produce, low fidelity.
- Claude can read the PDFs natively (`Read` tool, `pages` param) but that is page-ranged and
  ephemeral — not a durable, grep-able reference.
- Grammar is already authoritative in `vendor/SysML-v2-Release/bnf/KerML-textual-bnf.kebnf`.
  The abstract-syntax metamodel has a machine-readable XMI source at `omg.org/spec/KerML/20250201/`.

## Tool decision

- **Primary: Marker** (`marker-pdf`) — best code-block fidelity (the main pain), good image/table
  extraction, Apple MPS acceleration, optional `--use_llm` for messy pages. License: GPL (fine for
  internal reference; not redistributed commercially).
- **Compare: Docling** — strongest semantic structure + tables (TableFormer) + clean figure files.
- Decision rule: bake-off on the KerML PDF, then standardize the winner across the other PDFs.

Local uv venv on macOS is preferred over Docker for this one-time conversion (MPS acceleration; no
GPU-passthrough hassle). Docker only if we later want a repeatable service.

## Steps

1. **Isolated tooling env** (do NOT pollute the Rust workspace; keep venv out of git):
   - `uv venv` in a gitignored tools dir (e.g. `tools/pdfconv/`), Python 3.13.
   - `uv pip install marker-pdf docling` (expect a multi-GB model download on first run).
2. **Bake-off on KerML** (`vendor/SysML-v2-Release/doc/1-Kernel_Modeling_Language.pdf`):
   - Marker: `marker_single <pdf> --output_dir <out-marker>` (try `--use_llm` on a few hard pages).
   - Docling: `docling <pdf> --to md --output <out-docling>`.
   - Compare: code-block fencing, figure extraction (are diagrams saved as image files + linked?),
     table structure, heading hierarchy. Spot-check the abstract-syntax sections (§8.3.3.*).
3. **Pick winner**, then convert all four spec PDFs.
4. **Output layout** (finalize during session): one folder per spec, e.g.
   `docs/spec/<name>/<name>.md` + `docs/spec/<name>/figures/`. Replaces the manual `docs/spec/*.md`.
   Keep `docs/spec/kermlc/*.png` only if the converter does not already extract those figures.
5. **Commit** the generated Markdown + figures via a branch + PR (sizable doc addition). Update
   `CLAUDE.md` spec-reference note if the output layout changes.
